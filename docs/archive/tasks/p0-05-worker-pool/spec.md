# Spec: [P0-05] Worker pool and persistent job queue

**Slug:** p0-05-worker-pool **Issue:** #5 **Plan:** docs/archive/tasks/p0-05-worker-pool/plan.md **Date:** 2026-07-08

## Definition of Done

### Functional Requirements

- [ ] Given the renderer calls `jobs.enqueueDemo` with no input, when the handler/orchestrator runs, then it applies the demo defaults `totalSteps=10` and `stepMs=500` before repository enqueue, `repos.scanJobs.enqueue` inserts a row with `job_type='demo'`, `status='queued'`, `progress_current=0`, and `payload_json` containing those defaults, and the IPC call resolves `{ jobId }`.
- [ ] Given `jobs.enqueueDemo` is called with `{ totalSteps, stepMs }`, when the handler/orchestrator runs and both fields pass runtime validation, then it persists those caller-supplied values verbatim into `payload_json` instead of the defaults; the repository owns queue bookkeeping defaults only (`status`, `progress_current`, `priority`, `cancel_requested`), not demo-payload defaulting.
- [ ] Given the typed IPC request contract for `jobs.enqueueDemo`, when the runtime IPC validation rules are reviewed, then the accepted shape remains `void | { totalSteps?: number; stepMs?: number }`, with per-field validation on provided values only: `totalSteps` must be a finite integer between `1` and `1000` inclusive, `stepMs` must be a finite integer between `1` and `10000` inclusive, and omitted fields default to `10` and `500` respectively.
- [ ] Given `jobs.enqueueDemo` is called with exactly one of `totalSteps` or `stepMs` omitted and the other field valid, when the handler runs, then it preserves the valid provided field verbatim and applies the default only to the omitted field.
- [ ] Given `jobs.enqueueDemo` is called with an invalid `totalSteps` or `stepMs` at runtime (`NaN`, `Infinity`, `-Infinity`, non-integer, `< 1`, or above the inclusive max), when the handler validates the request, then the IPC call rejects before any orchestrator/repository enqueue occurs, the `scan_jobs` row count is unchanged, and the thrown error message names the offending field and its allowed inclusive range.
- [ ] Given a queued demo job and at least one idle worker slot, when the orchestrator's pump loop runs, then it calls `claimNext(workerId)`, the row transitions to `status='running'` with `worker_id` and `claimed_at` set and `started_at` set if previously unset, and the job is dispatched to a real `worker_threads` instance running `worker-entry.ts`.
- [ ] Given a demo job dispatched to a worker, when `runDemoJob` completes a step, then the worker posts a `ProgressMessage` over `postMessage`, the orchestrator persists it via `repo.updateProgress(id, { current, total, message })`, and the orchestrator's internal `onEvent` hook fires with a `JobProgressEvent` carrying exactly `{ jobId, jobType, status, current, total, message }`.
- [ ] Given an orchestrator `JobProgressEvent` reaches `main/index.ts`'s broadcast wiring, when it is mapped onto the typed IPC event surface, then `broadcast.ts` sends `jobs.progress` to every `BrowserWindow.getAllWindows()[].webContents` with the exact renderer-facing payload shape `{ id, jobType, status, progressCurrent, progressTotal, progressMessage, message }`, where `id=jobId`, `progressCurrent=current`, `progressTotal=total`, and `progressMessage=message`.
- [ ] Given the renderer's `JobDemo` component subscribed via `window.astrotracker.on('jobs.progress', listener)`, when a progress event for its own `jobId` arrives, then the displayed percentage/progress bar updates without any direct fs/db/IPC-channel access from renderer code outside the typed bridge.
- [ ] Given a demo job whose worker finishes its final step, when the worker posts a `DoneMessage`, then the orchestrator calls `repo.complete(id)`, the row reaches `status='completed'` with `finished_at` set and `worker_id` cleared, and the freed slot becomes available to the pump loop for the next claim.
- [ ] Given a demo job whose `scan_jobs` row is left `status='running'` because the previous app exit (crash or graceful quit) killed its worker, when the app restarts and `orchestrator.start()` runs, then `repo.requeueOrphaned()` resets that row to `status='queued'`, clearing `worker_id`/`claimed_at` while preserving `progress_current`, `progress_total`, `payload_json`, and `cancel_requested`.
- [ ] Given a requeued job with `progress_current > 0`, when it is reclaimed and dispatched to a fresh worker, then `runDemoJob` is invoked with `payload.resumeFrom` set from the persisted `progress_current`, and it reports progress only for its remaining steps (not restarting the progress sequence from 0).
- [ ] Given a demo job that is interrupted mid-run and then resumed after restart, when it eventually reaches `status='completed'`, then the combined ordered progress sequence observed across the pre-restart and post-restart orchestrator instances is exactly `1..payload.totalSteps` once each â€” exact count match, no duplicate step, no skipped step.
- [ ] Given `payload.resumeFrom` already equals `payload.totalSteps`, when `runDemoJob` is invoked on resume, then it emits no further progress events and returns normally without replaying any completed step.
- [ ] Given a still-queued (never-claimed) job, when `jobs.cancel({ jobId })` is called, then `repo.requestCancel` transitions the row directly to `status='cancelled'`, and the orchestrator does not attempt any pool-level cancel call for that job id (the pool never dispatched it).
- [ ] Given a running job, when `jobs.cancel({ jobId })` is called, then `repo.requestCancel` sets `cancel_requested=true` on the row (status stays `'running'`), the orchestrator forwards a `CancelMessage` to the worker currently running that job, and the worker's `ctx.isCancelled()` returns `true` at its next per-step check.
- [ ] Given a worker whose `ctx.isCancelled()` check trips, when it exits early, then it posts a `CancelledMessage`, and the orchestrator calls `repo.markCancelled(id)`, transitioning the row to the terminal `status='cancelled'` and freeing the slot for the next claim.
- [ ] Given a job already in a terminal state (`completed`/`failed`/`cancelled`), when `requestCancel`, `markCancelled`, `complete`, or `fail` is called on it again, then the row is unchanged and the call is a no-op, not a thrown error, enforced by a `WHERE status IN (...)` guard on the update statement rather than an application-level check-then-act read of current status.
- [ ] Given the app dies with a cancel in flight (cancel message sent to a worker, but its `CancelledMessage` ack never arrives before the process exits), when the app restarts, then `requeueOrphaned()` does **not** clear `cancel_requested`, and on reclaim the orchestrator checks `cancel_requested` before dispatching to a fresh worker â€” if true, it calls `markCancelled` directly instead of spinning up a worker just to cancel it.
- [ ] Given a worker process whose uncaught exception fires the pool's `worker.on('error', ...)` handler (a genuine crash, not a graceful `{type:'error'}` protocol message), when that happens mid-job, then the pool calls `repo.fail(jobId, error)` for the in-flight job and replaces the dead worker slot so pool capacity is not permanently reduced.
- [ ] Given a queued row is requeued on restart with `cancel_requested=true` before any new worker dispatch happens, when the orchestrator pump examines that row, then it marks the row `cancelled` directly and emits no worker progress for it â€” this restart-path cancellation is asserted separately from in-flight cancel acknowledgement.
- [ ] Given two jobs enqueued with equal `priority` and equal `created_at` (same-millisecond or explicitly seeded test data), when `claimNext` is called repeatedly, then rows are claimed by `created_at ASC` and then `id ASC` as the deterministic tie-breaker; any UUIDv7 monotonicity is helpful but not the acceptance-proof by itself.
- [ ] Given the pool has zero idle worker slots, when the orchestrator's pump loop runs, then it does not call `claimNext` at all â€” claim and dispatch are paired, so no row is ever left `status='running'` in the DB with no worker actually processing it.
- [ ] Given a job whose `progress_total` is `null` (indeterminate progress), when the `jobs.progress` event reaches the renderer, then `JobDemo` renders an indeterminate state and never divides by zero or `NaN`s the displayed percentage.
- [ ] Given more than one open `BrowserWindow`, when a progress event fires, then all open windows receive it (no per-window job-ownership filtering in this issue).
- [ ] Given `app.whenReady()`, when `main/index.ts` boots, then it opens the real `AstroDatabase` at `path.join(app.getPath('userData'), 'astrotracker.db')`, constructs a `WorkerPool` sized `Math.min(4, Math.max(1, os.cpus().length - 1))`, constructs the `JobQueueOrchestrator`, calls `orchestrator.start()`, and wires `orchestrator.onEvent` to the all-windows broadcast.
- [ ] Given the app's `before-quit` event fires, when the handler runs, then it calls `pool.terminateAll()` and `db.close()` on a best-effort basis (no throw blocks quit), relying on `requeueOrphaned()` at next boot for correctness regardless of shutdown cleanliness.
- [ ] Given the typed IPC request/response contract is declared in `packages/desktop/src/ipc/contract.ts`, when the required job procedures are added for the demonstrable workflow, then their exact signatures are: `jobs.enqueueDemo: void | { totalSteps?: number; stepMs?: number } -> { jobId: string }` and `jobs.cancel: { jobId: string } -> void`; `jobs.progress` remains the required renderer-observable event surface.

### Data Integrity

- [ ] `scan_jobs` gains `job_type TEXT NOT NULL`, `payload_json TEXT`, `progress_current INTEGER NOT NULL DEFAULT 0`, `progress_total INTEGER` (nullable â€” indeterminate), `progress_message TEXT`, `priority INTEGER NOT NULL DEFAULT 0`, `worker_id TEXT`, `claimed_at INTEGER` (`timestamp_ms`), `cancel_requested INTEGER NOT NULL DEFAULT 0`; `watch_folder_id` becomes nullable; created by migration `packages/db/drizzle/0002_*.sql`.
- [ ] `scan_jobs.status` gets a CHECK constraint restricting it to `'queued'|'running'|'completed'|'failed'|'cancelled'`, with default `'queued'` â€” Reviewer checks the generated migration's `CREATE TABLE __new_scan_jobs` for the CHECK clause, not just the Drizzle schema annotation.
- [ ] A composite index on `scan_jobs(status, priority, created_at)` exists, supporting the claim query â€” Reviewer confirms via `sqlite_master` `type='index'` or the migration SQL, and confirms `claimNext` actually orders queued rows by `priority DESC, created_at ASC, id ASC`, so equal-`created_at` ties resolve deterministically even though `id` is the final tie-breaker rather than part of the composite index.
- [ ] Migration 0002 is a table-rebuild (SQLite cannot ALTER a column to nullable or add a CHECK constraint in place): `CREATE __new_scan_jobs`, copy rows with `job_type` backfilled to `'scan'` for every pre-existing row, drop old table, rename, recreate indexes â€” Reviewer reads the generated SQL, not just the schema diff.
- [ ] Migration round-trips: `drizzle-kit generate` migration 0002 applies cleanly on an empty DB (fresh install, migrations 0000â†’0002 in sequence) and on a DB fixture seeded with pre-0002 `scan_jobs` rows (simulating an existing P0-04 install), and in the fixture case every pre-existing row ends up with `job_type='scan'` and its original `status`/`files_seen`/etc. columns intact.
- [ ] `payload_json` round-trips arbitrary valid job-shaped JSON (e.g. `{ totalSteps, stepMs, resumeFrom }` for `'demo'`) through `enqueue`/read without truncation or type coercion; invalid runtime `jobs.enqueueDemo` inputs are rejected at the IPC boundary before they reach repository persistence.
- [ ] `enqueue(input)` stamps `status='queued'`, `progress_current=0`, and sensible defaults for `priority` (0) and `cancel_requested` (0/false) when the caller omits them.
