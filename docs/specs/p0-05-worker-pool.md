# Spec: [P0-05] Worker pool and persistent job queue

**Slug:** p0-05-worker-pool **Issue:** #5 **Plan:** docs/plans/p0-05-worker-pool.md **Date:** 2026-07-08

## Definition of Done

### Functional Requirements

- [ ] Given the renderer calls `jobs.enqueueDemo` with no input, when the handler runs, then `repos.scanJobs.enqueue` inserts a row with `job_type='demo'`, `status='queued'`, `progress_current=0`, default `payload_json` (`totalSteps=10`, `stepMs=500`), and the IPC call resolves `{ jobId }`.
- [ ] Given `jobs.enqueueDemo` is called with `{ totalSteps, stepMs }`, when the handler runs, then those values are persisted verbatim into `payload_json` instead of the defaults.
- [ ] Given a queued demo job and at least one idle worker slot, when the orchestrator's pump loop runs, then it calls `claimNext(workerId)`, the row transitions to `status='running'` with `worker_id` and `claimed_at` set and `started_at` set if previously unset, and the job is dispatched to a real `worker_threads` instance running `worker-entry.ts`.
- [ ] Given a demo job dispatched to a worker, when `runDemoJob` completes a step, then the worker posts a `ProgressMessage` over `postMessage`, the orchestrator persists it via `repo.updateProgress(id, { current, total, message })`, and the `onEvent` hook fires with a `jobs.progress` event carrying `jobId`, `jobType`, `status`, `current`, `total`, `message`.
- [ ] Given a `jobs.progress` event fired by the orchestrator, when it reaches `main/index.ts`'s broadcast wiring, then `broadcast.ts` sends it to every `BrowserWindow.getAllWindows()[].webContents`, not a single tracked window.
- [ ] Given the renderer's `JobDemo` component subscribed via `window.astrotracker.on('jobs.progress', listener)`, when a progress event for its own `jobId` arrives, then the displayed percentage/progress bar updates without any direct fs/db/IPC-channel access from renderer code outside the typed bridge.
- [ ] Given a demo job whose worker finishes its final step, when the worker posts a `DoneMessage`, then the orchestrator calls `repo.complete(id)`, the row reaches `status='completed'` with `finished_at` set and `worker_id` cleared, and the freed slot becomes available to the pump loop for the next claim.
- [ ] Given a demo job whose `scan_jobs` row is left `status='running'` because the previous app exit (crash or graceful quit) killed its worker, when the app restarts and `orchestrator.start()` runs, then `repo.requeueOrphaned()` resets that row to `status='queued'`, clearing `worker_id`/`claimed_at` while preserving `progress_current`, `progress_total`, `payload_json`, and `cancel_requested`.
- [ ] Given a requeued job with `progress_current > 0`, when it is reclaimed and dispatched to a fresh worker, then `runDemoJob` is invoked with `payload.resumeFrom` set from the persisted `progress_current`, and it reports progress only for its remaining steps (not restarting the progress sequence from 0).
- [ ] Given a demo job that is interrupted mid-run and then resumed after restart, when it eventually reaches `status='completed'`, then the total count of progress ticks observed across the pre-restart and post-restart orchestrator instances equals `payload.totalSteps` exactly — no step double-counted, no step skipped.
- [ ] Given a still-queued (never-claimed) job, when `jobs.cancel({ jobId })` is called, then `repo.requestCancel` transitions the row directly to `status='cancelled'`, and the orchestrator does not attempt any pool-level cancel call for that job id (the pool never dispatched it).
- [ ] Given a running job, when `jobs.cancel({ jobId })` is called, then `repo.requestCancel` sets `cancel_requested=true` on the row (status stays `'running'`), the orchestrator forwards a `CancelMessage` to the worker currently running that job, and the worker's `ctx.isCancelled()` returns `true` at its next per-step check.
- [ ] Given a worker whose `ctx.isCancelled()` check trips, when it exits early, then it posts a `CancelledMessage`, and the orchestrator calls `repo.markCancelled(id)`, transitioning the row to the terminal `status='cancelled'` and freeing the slot for the next claim.
- [ ] Given a job already in a terminal state (`completed`/`failed`/`cancelled`), when `requestCancel`, `markCancelled`, `complete`, or `fail` is called on it again, then the row is unchanged and the call is a no-op, not a thrown error, enforced by a `WHERE status IN (...)` guard on the update statement rather than an application-level check-then-act read of current status.
- [ ] Given the app dies with a cancel in flight (cancel message sent to a worker, but its `CancelledMessage` ack never arrives before the process exits), when the app restarts, then `requeueOrphaned()` does **not** clear `cancel_requested`, and on reclaim the orchestrator checks `cancel_requested` before dispatching to a fresh worker — if true, it calls `markCancelled` directly instead of spinning up a worker just to cancel it.
- [ ] Given a worker process whose uncaught exception fires the pool's `worker.on('error', ...)` handler (a genuine crash, not a graceful `{type:'error'}` protocol message), when that happens mid-job, then the pool calls `repo.fail(jobId, error)` for the in-flight job and replaces the dead worker slot so pool capacity is not permanently reduced.
- [ ] Given two jobs enqueued with equal `priority` (including two enqueued within the same millisecond), when `claimNext` is called repeatedly, then rows are claimed in `created_at` order (oldest first), using UUIDv7's monotonic same-millisecond ordering to stay deterministic.
- [ ] Given the pool has zero idle worker slots, when the orchestrator's pump loop runs, then it does not call `claimNext` at all — claim and dispatch are paired, so no row is ever left `status='running'` in the DB with no worker actually processing it.
- [ ] Given a job whose `progress_total` is `null` (indeterminate progress), when the `jobs.progress` event reaches the renderer, then `JobDemo` renders an indeterminate state and never divides by zero or `NaN`s the displayed percentage.
- [ ] Given more than one open `BrowserWindow`, when a progress event fires, then all open windows receive it (no per-window job-ownership filtering in this issue).
- [ ] Given `app.whenReady()`, when `main/index.ts` boots, then it opens the real `AstroDatabase` at `path.join(app.getPath('userData'), 'astrotracker.db')`, constructs a `WorkerPool` sized `Math.min(4, Math.max(1, os.cpus().length - 1))`, constructs the `JobQueueOrchestrator`, calls `orchestrator.start()`, and wires `orchestrator.onEvent` to the all-windows broadcast.
- [ ] Given the app's `before-quit` event fires, when the handler runs, then it calls `pool.terminateAll()` and `db.close()` on a best-effort basis (no throw blocks quit), relying on `requeueOrphaned()` at next boot for correctness regardless of shutdown cleanliness.
- [ ] Given `jobs.list` is invoked, when the handler runs, then it returns a summary of jobs (at minimum id, jobType, status, progress fields) sourced from the repository, with no pagination/filtering UI required (Out of Scope).

### Data Integrity

- [ ] `scan_jobs` gains `job_type TEXT NOT NULL`, `payload_json TEXT`, `progress_current INTEGER NOT NULL DEFAULT 0`, `progress_total INTEGER` (nullable — indeterminate), `progress_message TEXT`, `priority INTEGER NOT NULL DEFAULT 0`, `worker_id TEXT`, `claimed_at INTEGER` (`timestamp_ms`), `cancel_requested INTEGER NOT NULL DEFAULT 0`; `watch_folder_id` becomes nullable; created by migration `packages/db/drizzle/0002_*.sql`.
- [ ] `scan_jobs.status` gets a CHECK constraint restricting it to `'queued'|'running'|'completed'|'failed'|'cancelled'`, with default `'queued'` — Reviewer checks the generated migration's `CREATE TABLE __new_scan_jobs` for the CHECK clause, not just the Drizzle schema annotation.
- [ ] A composite index on `scan_jobs(status, priority, created_at)` exists, supporting the claim query — Reviewer confirms via `sqlite_master` `type='index'` or the migration SQL, and confirms `claimNext`'s query actually uses `status`, `priority`, and `created_at` in its `WHERE`/`ORDER BY` (an index nothing queries against is not sufficient).
- [ ] Migration 0002 is a table-rebuild (SQLite cannot ALTER a column to nullable or add a CHECK constraint in place): `CREATE __new_scan_jobs`, copy rows with `job_type` backfilled to `'scan'` for every pre-existing row, drop old table, rename, recreate indexes — Reviewer reads the generated SQL, not just the schema diff.
- [ ] Migration round-trips: `drizzle-kit generate` migration 0002 applies cleanly on an empty DB (fresh install, migrations 0000→0002 in sequence) and on a DB fixture seeded with pre-0002 `scan_jobs` rows (simulating an existing P0-04 install), and in the fixture case every pre-existing row ends up with `job_type='scan'` and its original `status`/`files_seen`/etc. columns intact.
- [ ] `payload_json` round-trips arbitrary job-shaped JSON (e.g. `{ totalSteps, stepMs, resumeFrom }` for `'demo'`) through `enqueue`/read without truncation or type coercion — validated as parseable JSON before write, not merely stored as opaque text with no validation at all.
- [ ] `enqueue(input)` stamps `status='queued'`, `progress_current=0`, and sensible defaults for `priority` (0) and `cancel_requested` (0/false) when the caller omits them.

### Documented Deviations (review callouts — confirm justified, not silent drift)

These are the plan's 9 flagged defaults chosen where no DD dictates the answer. Reviewer must confirm each is (a) actually implemented as described and (b) called out in the PR description, not left implicit:

- [ ] **(1) The job queue is `scan_jobs`, generalized — not a new table.** Confirm no new `job_queue`/`jobs` table was introduced instead; the generic queue columns live on `scan_jobs` alongside the untouched scan-specific columns (`files_seen`/`files_added`/`files_updated`).
- [ ] **(2) `job_type` is unconstrained `TEXT NOT NULL`, no CHECK/enum.** Confirm the migration's CHECK constraint covers `status` only, not `job_type`; confirm the `JobType` union (`'demo'` today) is defined in `packages/desktop` (e.g. `protocol.ts`), not in `packages/db` — `packages/db` only knows `status`.
- [ ] **(3) Workers never touch SQLite — main process is the sole DB writer.** Confirm `worker-entry.ts`, `demo-job.ts`, and `pool.ts` import no `@astrotracker/db`, no `better-sqlite3`, and open no database connection; Reviewer greps `packages/desktop/src/main/jobs/*.ts` (excluding `orchestrator.ts`) for `astrotracker/db` / `better-sqlite3` imports and confirms none exist outside `orchestrator.ts`.
- [ ] **(4) No automatic retry-on-failure.** Confirm no `attempts`/`max_attempts`/backoff columns or logic exist anywhere in the diff; a failed job goes straight to terminal `status='failed'` with no re-queue; only `requeueOrphaned()` (a distinct restart-recovery mechanism, not retry) puts a row back in `'queued'`.
- [ ] **(5) Worker pool size is `Math.min(4, Math.max(1, os.cpus().length - 1))`, computed once at startup.** Reviewer confirms this exact formula (or an equivalent producing identical results) appears in `main/index.ts` and is passed into the `WorkerPool` constructor — not hardcoded to a fixed number, not read from settings/UI.
- [ ] **(6) Cancellation is cooperative, checked once per step — not preemptive mid-sleep.** Confirm `runDemoJob` calls `ctx.isCancelled()` between steps (not inside a step's `setTimeout`/sleep), and no `AbortController`-wrapped per-tick cancellation exists — the demo job's default 10-step granularity is the acceptance bar for responsiveness, not sub-step interruption.
- [ ] **(7) Worker script bundling uses electron-vite's `?modulePath` import suffix.** Confirm `worker-entry.ts` is referenced via `import workerEntryPath from './worker-entry?modulePath'` (or equivalent) in `pool.ts`, and `electron.vite.config.ts` is unmodified (per plan, no manual multi-entry config needed) — Reviewer flags any hand-rolled `electron.vite.config.ts` worker entry as an unexplained deviation from the documented default.
- [ ] **(8) `@astrotracker/db` moves from `devDependencies` to `dependencies` in `packages/desktop/package.json`.** Confirm the diff shows this move and confirm no other package's dependency placement changed incidentally.
- [ ] **(9) Progress events broadcast to every open `BrowserWindow`, not a tracked originating window.** Confirm `broadcast.ts`'s `IpcSenderLike` iterates `BrowserWindow.getAllWindows()` rather than a single stored window reference; Reviewer confirms there is no per-window/per-job routing table in this diff (that's explicitly out of scope).

- [ ] New domain logic is in packages/core with no Electron/fs imports — N/A narrowly: this issue adds no `packages/core` code (the demo job and orchestrator are Electron/infra plumbing per DD-002 rule 1, correctly placed in `packages/desktop`); Reviewer confirms `packages/core` is untouched by this diff.
- [ ] All persisted timestamps are UTC — **DD-003 hard rule, explicit check:** `scan_jobs.claimed_at`, `.started_at`, `.finished_at` all use the existing `timestamp_ms` Drizzle column mode (epoch-ms, UTC by construction, matching P0-04's convention); Reviewer checks the new columns in `packages/db/src/schema/infra.ts` use `integer(..., { mode: 'timestamp_ms' })`, not a TEXT date string.
- [ ] **DD-003 hard rule, explicit check — UUIDv7 PK + `updated_at` on `scan_jobs`:** the migration must not alter `scan_jobs.id` (stays UUIDv7, populated by `@astrotracker/core`'s `uuidv7()`) or drop `updated_at`; every new queue-primitive write helper (`enqueue`, `claimNext`, `updateProgress`, `requestCancel`, `markCancelled`, `complete`, `fail`, `requeueOrphaned`) re-stamps `updated_at` — Reviewer checks each method in `scan-jobs.ts` for the stamp call, same pattern as P0-04's `repositories/shared.ts` helper.
- [ ] Manual user overrides (target/filter/type/session assignments) survive a rescan — N/A, this issue introduces no scanning or rescan logic (P1-06 is out of scope); the demo job touches no `frames`/`targets` rows.
- [ ] Repositories expose no raw connection to callers — the new `scan-jobs.ts` queue primitives return typed row/void values only, never a `better-sqlite3` `Database`/`Statement`; Reviewer spot-checks `claimNext`'s return type is the row shape, not a prepared statement or transaction handle.

## Out of Scope

Copied and expanded from the plan — Reviewer must not flag any of the following as missing from this PR:

- Real scan/hash/thumbnail job types (P1-06, P1-08, Phase 1 thumbnail work) — only `'demo'` ships; the message protocol (`protocol.ts`) and worker registry (`{ demo: runDemoJob }`) are shaped so adding a type is additive, not a review requirement to prove extensibility beyond the shape itself.
- Retry-on-failure / backoff policy (`attempts`/`max_attempts`) — failed jobs are terminal; do not flag the absence of retry as a gap.
- Heartbeat-based detection of a hung-but-alive worker (distinct from an app restart) — only protocol-level `'error'` events and boot-time orphan requeue are handled; a worker that hangs forever without the app restarting is explicitly undetected in this issue.
- Job history/pagination UI, priority-setting UI — `jobs.list` is a minimal debug surface, not designed UX (DD-008 UX work is Phase 1).
- chokidar/watch-folder scanning itself (P1-06) — this issue only builds the queue P1-06 will enqueue into; `watch_folder_id` being nullable is the only scanning-adjacent schema change.
- CI benchmark gates (P0-07) — see Performance section.
- Free-tier limits, cloud sync (Phase 2).
- Multi-process DB contention beyond P0-04's existing `busy_timeout` — no new concern introduced; the main process remains the sole writer.
- Exact SQL formatting/naming of the generated migration file beyond matching the schema module (Reviewer should diff behavior — CHECK constraint, nullable column, index, backfill — not bikeshed Drizzle's auto-generated table-rebuild naming).
- Window-scoped/per-job progress routing (Default 9) — broadcast-to-all is correct for this issue; do not flag the lack of per-window filtering.
- Settings UI for pool size (Default 5) — the formula is a constant computed at startup, not a configurable surface.

## Test Hints

- **enqueue-defaults**: call `repos.scanJobs.enqueue({ jobType: 'demo' })` with no payload; assert `status='queued'`, `progressCurrent=0`, `priority=0`, `cancelRequested=false`, `payloadJson` reflects the default totalSteps/stepMs applied by the demo-job caller layer (or repository default, per implementation).
- **claim-priority-fifo**: enqueue three jobs — two `priority=0` (A then B, same millisecond if possible), one `priority=1` (C) — enqueued last; assert `claimNext` order is C, A, B.
- **claim-empty-queue**: call `claimNext('worker-1')` on an empty `scan_jobs` table; assert it returns `undefined`/no row, no throw.
- **requeue-preserves-progress-and-cancel**: seed a `'running'` row with `progressCurrent=4`, `progressTotal=10`, `cancelRequested=true`; call `requeueOrphaned()`; assert `status='queued'`, `workerId`/`claimedAt` cleared, `progressCurrent`/`progressTotal`/`cancelRequested` unchanged.
- **cancel-queued-vs-running**: enqueue job A (leave queued), enqueue+claim job B (`'running'`); call `requestCancel` on both; assert A is immediately `'cancelled'`, B stays `'running'` with `cancelRequested=true`.
- **terminal-no-op**: seed a `'completed'` row; call `requestCancel`, `markCancelled`, `complete`, `fail` on it in turn; assert the row (status, timestamps) is unchanged after each call and none throws.
- **demo-job-progress-sequence**: run `runDemoJob({ totalSteps: 5, stepMs: 1 }, ctx)` with a recording `ctx.reportProgress`; assert five calls with `current` 1..5, `total=5`.
- **demo-job-resume-from**: run `runDemoJob({ totalSteps: 5, stepMs: 1, resumeFrom: 3 }, ctx)`; assert `reportProgress` is called only for steps 4 and 5, not 1-3.
- **demo-job-cancel-mid-run**: run `runDemoJob` with a `ctx.isCancelled()` stub that flips `true` after the 2nd step; assert the function exits after reporting step 2, no further steps run.
- **pool-worker-error-recovery**: dispatch a job to the pool, force the worker's `'error'` event (e.g. inject a throwing job type or kill the thread), assert the in-flight job is failed and the pool still has full capacity for a subsequent dispatch.
- **orchestrator-restart-resume**: enqueue a demo job with `totalSteps: 20, stepMs: 20`; wait for a progress-count promise to resolve at `current >= 5`; terminate the orchestrator/pool without graceful completion; open a second orchestrator/pool against the same DB file; call `start()`; assert the job resumes and completes, and the sum of `reportProgress` calls observed pre- and post-restart equals 20 with no duplicate/skip.
- **orchestrator-cancel-flow**: enqueue a longer demo job, cancel shortly after `'running'`, assert final `status='cancelled'` and that a subsequently queued job is picked up on the freed slot.
- **ipc-event-whitelist**: call the `create-listen.ts` factory with a rejected channel name (e.g. `'evil.channel'`); assert it throws before calling the raw `ipcRenderer.on`; call it with `'jobs.progress'`; assert it forwards and returns a working unsubscribe function.
- **job-demo-component**: render `JobDemo` with mocked `window.astrotracker`; click enqueue; assert `invoke('jobs.enqueueDemo', ...)` called; fire a mocked `'jobs.progress'` event via the captured `on` listener; assert displayed percentage updates; unmount; assert the unsubscribe function returned by `on` was called.
