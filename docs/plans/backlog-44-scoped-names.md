# Plan: [Backlog] Scope package names to @astrotracker/* before real deps land

**Slug:** backlog-44-scoped-names   **Issue:** #44   **Date:** 2026-07-05
**Governing DDs:** DD-002 (application architecture / package layout ‚Äî layering and dependency
direction are unaffected by this rename)
**Status:** READY_FOR_SPEC

## Summary

The four workspace packages currently publish under bare npm names (`core`, `db`, `desktop`,
`renderer`) that are collision-prone once real third-party dependencies land. This is a
purely mechanical rename to scoped names (`@astrotracker/core`, `@astrotracker/db`,
`@astrotracker/desktop`, `@astrotracker/renderer`) ‚Äî no behavior, layering, or dependency
direction changes. Directory paths (`packages/core`, `packages/desktop/renderer`, etc.) are
untouched; only the `name` field in each `package.json`, the `workspace:*` dependency
specifiers that reference those names, the bare-specifier imports in source that resolve
through Node module resolution (`from 'core'`, `from 'db'`), and the regenerated
`pnpm-lock.yaml` change. Vitest project names and the ESLint config need no changes, since
both key off directory paths, not package names ‚Äî confirmed by inspection below.

## Affected Files

- `packages/core/package.json` ‚Äî `name: "core"` ‚Üí `"@astrotracker/core"`
- `packages/db/package.json` ‚Äî `name: "db"` ‚Üí `"@astrotracker/db"`; `dependencies.core` key
  ‚Üí `dependencies["@astrotracker/core"]` (value stays `workspace:*`)
- `packages/desktop/package.json` ‚Äî `name: "desktop"` ‚Üí `"@astrotracker/desktop"`;
  `dependencies.core`/`dependencies.db` keys ‚Üí `@astrotracker/core`/`@astrotracker/db`
- `packages/desktop/renderer/package.json` ‚Äî `name: "renderer"` ‚Üí `"@astrotracker/renderer"`
- `packages/db/src/index.ts` ‚Äî `import { coreVersion } from 'core'` ‚Üí
  `from '@astrotracker/core'`
- `packages/desktop/src/index.ts` ‚Äî `import { coreVersion } from 'core'` and
  `import { dbVersion } from 'db'` ‚Üí `@astrotracker/core` / `@astrotracker/db`
- `pnpm-lock.yaml` ‚Äî regenerated by `pnpm install` (importer dependency keys/specifiers
  update to the scoped names; importer paths themselves, e.g. `packages/db:`, are unaffected)
- `README.md` ‚Äî package-layout table text that names dependencies as `core`, `db` (lines ~30‚Äì31)
  updates to `@astrotracker/core`, `@astrotracker/db`; path references (`packages/core`, etc.)
  are unchanged since directories don't move

Not touched (verified, reasons noted):
- `vitest.workspace.ts` ‚Äî project `name` fields (`'core'`, `'db'`, `'desktop'`, `'renderer'`)
  are Vitest project labels keyed to `root` paths, not npm package names; no collision risk,
  no functional coupling to `package.json#name`. Left as-is to keep the diff minimal ‚Äî renaming
  them is optional polish, not required by the issue's collision concern.
- `eslint.config.mjs` ‚Äî scopes its `no-restricted-imports` rule via the file glob
  `packages/core/src/**/*.ts` (a path), not a package name. No change needed.
- `tsconfig.base.json` / per-package `tsconfig.json` ‚Äî no TS project references or path
  mappings key off package names (each just `extends` the base by relative path).
- `docs/plans/p0-01-monorepo-init.md`, `docs/specs/p0-01-monorepo-init.md` ‚Äî historical
  record of already-completed issue P0-01; they document what the dependency graph looked
  like at that time using the old bare names. Left as-is (see Out of Scope).

## Implementation Steps

### Step 1 ‚Äî Rename package identities in the four `package.json` files
- **Outcome:** Each of the four workspace packages declares its scoped name
  (`@astrotracker/core`, `@astrotracker/db`, `@astrotracker/desktop`, `@astrotracker/renderer`)
  as `package.json#name`. Internal `workspace:*` dependency entries in `packages/db/package.json`
  and `packages/desktop/package.json` are re-keyed to the scoped names of the packages they
  depend on (`db` depends on `@astrotracker/core`; `desktop` depends on `@astrotracker/core`
  and `@astrotracker/db`). The `workspace:*` version specifier itself is unchanged.
- **Files:** `packages/core/package.json`, `packages/db/package.json`,
  `packages/desktop/package.json`, `packages/desktop/renderer/package.json`
- **Depends on:** none

### Step 2 ‚Äî Update source imports to the scoped specifiers
- **Outcome:** `packages/db/src/index.ts` and `packages/desktop/src/index.ts` import from
  `@astrotracker/core` (and `@astrotracker/db` in desktop's case) instead of the bare `core`/`db`
  specifiers, so the code resolves against the renamed packages once dependencies are
  reinstalled.
- **Files:** `packages/db/src/index.ts`, `packages/desktop/src/index.ts`
- **Depends on:** Step 1

### Step 3 ‚Äî Regenerate the lockfile
- **Outcome:** `pnpm-lock.yaml` reflects the new package names and workspace links with no
  drift; `pnpm install` completes without needing `--no-frozen-lockfile` or manual edits.
- **Files:** `pnpm-lock.yaml` (regenerated, not hand-edited)
- **Depends on:** Step 1 (run `pnpm install` from repo root after the `package.json` renames)

### Step 4 ‚Äî Update README package-layout references
- **Outcome:** The package-layout table and prose in `README.md` describe dependencies using
  the scoped names (`@astrotracker/core`, `@astrotracker/db`) so the doc matches the actual
  `package.json` dependency declarations. Directory-path references stay as `packages/core`,
  `packages/db`, etc. ‚Äî those don't change.
- **Files:** `README.md`
- **Depends on:** Step 1

### Step 5 ‚Äî Verify the gate
- **Outcome:** `pnpm install && pnpm -r build && pnpm -r lint && pnpm -r test` all pass,
  confirming the rename didn't break module resolution, the `core`-purity ESLint rule (still
  path-scoped, unaffected), or the placeholder unit tests (which assert on `coreVersion`/
  `dbVersion` values and `describe*()` string output, not on package names ‚Äî unaffected by
  the rename).
- **Files:** none (verification only)
- **Depends on:** Steps 1‚Äì4

## Edge Cases

- **Stale `node_modules` symlinks:** existing `node_modules/core`, `node_modules/db` symlinks
  from the old bare names could linger and shadow the new scoped resolution if `pnpm install`
  is run without a clean workspace. The plan's gate command runs `pnpm install` first each
  time, which pnpm handles correctly (it prunes stale workspace links), but if the Coder finds
  leftover bare-name directories under `node_modules/` after install, remove `node_modules` at
  the affected package(s) and reinstall rather than leaving both old and new links present.
- **Partial rename causing a silent bare-specifier fallback:** if `packages/db/src/index.ts`
  is updated to import `@astrotracker/core` but `packages/db/package.json`'s dependency key is
  left as `core`, `pnpm install` won't create the `@astrotracker/core` link and the build fails
  with a clear module-not-found error rather than resolving to something unintended ‚Äî so Steps
  1 and 2 must land together, not independently, before running install.
- **`workspace:*` value vs. key:** only the dependency *key* (package name) changes; the
  `workspace:*` specifier *value* must NOT be rewritten to `workspace:@astrotracker/core:*` or
  similar ‚Äî `workspace:*` is a protocol specifier keyed by the dependency name itself.

## Out of Scope

- Renaming the Vitest project names in `vitest.workspace.ts` (`'core'`, `'db'`, `'desktop'`,
  `'renderer'`) ‚Äî these are display labels tied to `root` paths, not collision-prone npm
  identifiers; leaving them unscoped carries no risk and renaming is cosmetic.
- Editing `docs/plans/p0-01-monorepo-init.md` or `docs/specs/p0-01-monorepo-init.md` to
  reflect the new names ‚Äî these are historical records of a already-completed, already-merged
  issue (P0-01) and describe the state of the repo as it was at that time; rewriting history
  docs is not part of this rename.
- Adding a `packages/cloud` package or any Phase 2 cloud-architecture work (DD-007) ‚Äî unrelated
  to this issue.
- Any actual new runtime dependency being added to any package ‚Äî this issue is preparatory
  scoping only, per its title ("before real deps land").
- Changing package `version` fields, `description` fields, or any field other than `name` and
  the `dependencies` keys that reference the renamed packages.
